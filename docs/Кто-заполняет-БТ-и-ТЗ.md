# Кто что заполняет в БТ и ТЗ

Разбор документов [Бизнес-требования](Бизнес-требования-шаблон.md) и [ТЗ](ТЗ-шаблон.md) по ответственности за заполнение разделов. Роли: **Продукт/проджект (PM)**, **Заказчик**, **Аналитик**, **Разработчик**.

---

## Документ бизнес-требований (БТ)

| Раздел | Кто заполняет | Что делает | Примечание |
|--------|----------------|------------|------------|
| **1. Бизнес-контекст** | | | |
| 1.1 Название продукта | PM | Вносит согласованное название. | |
| 1.2 Проблема / возможность | PM | Формулирует проблему или возможность. | Может опереться на интервью с заказчиком или пользователями. |
| 1.3 Ценность для пользователя | PM | Перечисляет выгоды для конечного пользователя. | Заказчик может подсказать «боли» пользователей. |
| 1.4 Ценность для заказчика | PM | Описывает выгоды для заказчика/владельца продукта. | Желательно согласовать с заказчиком. |
| **2. Заинтересованные стороны** | PM | Заполняет таблицу ролей, описаний и интересов. | При необходимости уточняет у заказчика состав стейкхолдеров. |
| **3. Цели продукта** | PM | Формулирует цели и критерии достижения. | Согласование с заказчиком; цели без технических деталей. |
| **4. Пользователи и сценарии** | PM или Аналитик | Портрет пользователя и таблица сценариев (действие → результат). | PM может делегировать аналитику; заказчик/пользователи — источник сценариев. |
| **5. Область проекта (scope)** | PM | Список «в границах» и «вне границ». | Обязательно согласовать с заказчиком, чтобы избежать размывания scope. |
| **6. Критерии успеха** | PM | Таблица критериев и способов проверки. | С точки зрения результата для пользователя/заказчика; согласовать с заказчиком. |
| **7. Ограничения и допущения** | PM | Ограничения (что не гарантируем) и допущения (что считаем истиной). | Разработчик может подсказать технические допущения и ограничения. |
| **8. Риски и митигация** | PM | Таблица рисков, влияния, вероятности и митигации. | Разработчик может добавить технические риски (интеграции, производительность). |
| **9. Глоссарий** | PM или Аналитик | Определения терминов предметной области. | Чтобы заказчик и разработчик использовали термины одинаково. |
| **10. Связь с другими документами** | Не требуется | Раздел-справка; ссылки обновлять при появлении новых документов. | |

**Итог по БТ:** основной владелец документа — **PM**. Заказчик участвует в согласовании целей, scope и критериев успеха. Аналитик (если есть) может детализировать сценарии и глоссарий. Разработчик привлекается точечно (допущения, риски).

## Техническое задание (ТЗ)

| Раздел | Кто заполняет | Что делает | Примечание |
|--------|----------------|------------|------------|
| **1. Общие сведения** | PM | Название, назначение, целевая платформа. | Назначение — кратко из БТ; платформу можно уточнить с разработчиком. |
| **2. Цели и границы проекта** | PM | Списки «в рамках» и «вне рамок». | Перенос из БТ (раздел scope) в формулировках, пригодных для разработки. |
| **3. Ролевая модель и пользователи** | PM | Таблица ролей и описаний. | Соответствует заинтересованным сторонам и пользователям из БТ; без технических деталей прав. |
| **4. Функциональные требования** | PM + Разработчик | **PM:** колонка «Требование» (что нужно). **Разработчик:** колонка «Реализация» (как сделано/будет сделано — элементы UI, вызовы API). | Требования не должны предписывать конкретный стек; реализация уточняется по ходу разработки. |
| **5. Нефункциональные требования** | PM + Разработчик | **PM:** формулирует требование (например, «секреты не в репозитории»). **Разработчик:** как реализовано (например, .env, .gitignore). | |
| **6. Технологический стек** | Разработчик | Предлагает и заполняет таблицу (backend, frontend, БД, сервисы, конфигурация). | PM утверждает или согласовывает ограничения (бюджет, корпоративные стандарты). |
| **7. Модель данных (БД)** | Разработчик | Таблицы, поля, типы, ограничения, индексы. | Должна обеспечивать сценарии из БТ; PM проверяет, что сущностей и полей достаточно. |
| **8. API (контракт backend)** | Разработчик | Методы, URL, параметры, тело запроса, ответы, коды ошибок. | Должен покрывать функциональные требования; PM проверяет соответствие сценариям. |
| **9. Структура приложения (архитектура)** | Разработчик | Описание слоёв backend и frontend, конфигурация. | Для согласованности с реализацией и онбординга новых разработчиков. |
| **10. Критерии приёмки (чек-лист)** | PM | Список проверок для приёмки. | На основе целей и сценариев из БТ и требований из ТЗ; можно согласовать с заказчиком. |
| **11. Приложения и ссылки** | Не требуется / Разработчик | Ссылки на README, БТ, план, схему БД, ключевые файлы. | Обновлять при появлении новых артефактов. |

**Итог по ТЗ:** **PM** заполняет общие сведения, цели и границы, роли, формулировки функциональных и нефункциональных требований и критерии приёмки. **Разработчик** заполняет стек, модель данных, контракт API, архитектуру и колонку «Реализация» в таблицах требований. Согласование совместное: ТЗ не должен противоречить БТ, реализация — покрывать все требования.

---

## Порядок заполнения (рекомендуемый)

1. **PM** готовит черновик БТ (разделы 1–6, при необходимости 7–9).
2. **Заказчик** согласовывает цели, scope и критерии успеха в БТ.
3. **PM** переносит согласованное в черновик ТЗ (разделы 1–3, 4–5 — колонка «Требование», раздел 10).
4. **Разработчик** предлагает стек (раздел 6), модель данных (7), контракт API (8), архитектуру (9) и колонку «Реализация» в разделах 4–5.
5. **PM** проверяет соответствие ТЗ БТ и при необходимости правит требования или согласовывает изменения с заказчиком.
6. После утверждения ТЗ разработка ведётся по ТЗ; при приёмке используется чек-лист раздела 10 (ответственный за проверку — PM).

## Краткая сводка по ролям

| Роль | В БТ | В ТЗ |
|------|------|------|
| **PM** | Владелец документа: контекст, цели, сценарии, scope, критерии успеха, ограничения, риски, глоссарий. | Общие сведения, границы, роли, формулировки требований, критерии приёмки. |
| **Заказчик** | Согласование целей, scope, критериев успеха. | Не заполняет; утверждает результат вместе с PM. |
| **Аналитик** | Может детализировать сценарии и глоссарий. | — |
| **Разработчик** | Точечно: допущения, технические риски. | Стек, модель данных, API, архитектура, колонка «Реализация» в требованиях. |
